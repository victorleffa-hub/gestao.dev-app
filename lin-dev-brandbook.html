01

Visão Geral da Marca

O que é o Lin-Dev

O Lin-Dev é um project manager construído exclusivamente para desenvolvedores e squads de desenvolvimento. Diferente de ferramentas de gestão genéricas, ele foi desenhado para devolver ao time o tempo perdido com tarefas operacionais — atualizar boards, mover cards, preencher status, re-explicar contexto.

A ferramenta cuida da burocracia de gestão para que o dev cuide do código. Sprints, tarefas, ideias e bugs vivem num lugar só, com dashboards que mostram exatamente o que está acontecendo — quem está fazendo o quê, o que está atrasado, onde o time está travado — sem precisar perguntar para ninguém.

Além disso, possui integração nativa com o protocolo MCP (Model Context Protocol), permitindo que ferramentas de AI coding acessem tarefas e contexto diretamente no terminal. Mas a IA é meio, não fim: o produto é a gestão fluida, não o hype de inteligência artificial.

Categoria de mercado

Project Management para Desenvolvedores — uma subcategoria que o Lin-Dev está definindo. Não competimos com Jira, Linear ou Asana por features de PM genérico. Competimos com o fato de que nenhuma ferramenta de gestão existente foi construída para se encaixar no dia a dia real de quem coda.

Diferenciais inegáveis

Board que se atualiza

O squad para de gastar 25+ min/dia arrastando cards. A ferramenta mantém o board em sincronia com o que está acontecendo de verdade.

Dashboards de clareza total

Visão em tempo real de prazos, velocidade do time, tarefas bloqueadas e distribuição de carga. Leitura instantânea.

MCP nativo

O Claude Code (ou qualquer ferramenta compatível) acessa tarefas direto no terminal. Zero copy-paste.

Dev Virtual

Agente autônomo que pega tarefas do kanban, implementa e abre PRs — paga apenas quando trabalha.

Workflow Ideia→Exec

De uma ideia no banco de ideias até um PR aberto, sem fricções.

Setup em 2 minutos

Zero configuração complexa. Criou conta, montou o board, pronto.

02

Propósito, Missão, Visão e Valores

Propósito — Por que existimos

Devolver ao desenvolvedor o tempo que ferramentas de gestão roubam dele todos os dias.

Toda hora gasta arrastando cards, preenchendo status e montando relatórios manuais é uma hora roubada da criação. Existimos para eliminar essa burocracia e dar ao dev visão clara do que importa — sem esforço.

Missão — O que fazemos

Simplificar radicalmente a gestão de projetos de desenvolvimento, automatizando o trabalho operacional e entregando dashboards que transformam dados dispersos em decisões claras — para que o time foque em resolver problemas, não em manter ferramentas.

Visão — Onde queremos chegar

Ser o project manager padrão de squads de desenvolvimento em todo o mundo — a ferramenta que finalmente fez a gestão desaparecer dentro do fluxo de trabalho.

Valores da marca

ValorSignificado prático
ClarezaO time precisa ver o que está acontecendo em 3 segundos. Dashboards legíveis, dados em tempo real, zero ambiguidade sobre prazos, responsáveis e bloqueios.
FluidezTudo deve fluir. Se o dev precisa trocar de aba ou gastar tempo atualizando board, a ferramenta falhou.
PrecisãoFuncionalidade cirúrgica. Zero feature bloat. Cada elemento resolve um problema real de quem desenvolve.
TransparênciaPreço claro, roadmap aberto, comunicação honesta. Sem asteriscos.

03

Posicionamento Estratégico

Declaração de posicionamento

Para desenvolvedores e squads de desenvolvimento que perdem tempo gerenciando ferramentas em vez de codar, o Lin-Dev é o project manager que cuida da burocracia de gestão sozinho — mantendo o board atualizado, mostrando prazos e bloqueios em dashboards claras, e conectando tarefas ao terminal via MCP — para que o time gaste zero tempo organizando e 100% do tempo criando.

Framework de posicionamento

DimensãoDefinição
Público-alvoDevs solo, squads de 2-15 desenvolvedores, founders técnicos, tech leads de startups
CategoriaProject Management para Desenvolvedores
Promessa centralO board se atualiza, as dashboards mostram tudo, e o time ganha horas de volta todo dia
Razão para acreditarAutomação real do board + dashboards em tempo real + MCP nativo + Dev Virtual
Território emocionalO alívio de abrir o board e ele já estar certo. Sem ter feito nada.
InimigoO tempo perdido com burocracia de gestão: arrastar cards, preencher status, montar relatórios, re-explicar contexto.

Diferenciação competitiva

O Lin-Dev não compete por features de PM genérico. Compete por relevância no dia a dia do desenvolvedor. Ferramentas tradicionais exigem que o dev pare de codar para gerenciar. O Lin-Dev faz a gestão acontecer automaticamente, mostra o status real via dashboards e devolve ao time o tempo que antes era gasto em trabalho operacional.

A conexão com IA (MCP) é uma consequência natural dessa filosofia: se a ferramenta já organiza o contexto de forma estruturada, entregar isso para um agente de IA no terminal é o próximo passo lógico — não o ponto de partida.

05

Arquétipo de Marca

Primário: O Mago (The Magician)

O Mago transforma a realidade. Pega algo complexo e faz parecer simples. É exatamente o que o Lin-Dev faz: transforma a gestão de projetos de um fardo burocrático em algo que simplesmente acontece — o board se atualiza, as dashboards se montam, o dev nunca precisou parar de codar.

  • Motivação: Entender as leis fundamentais e usá-las para criar transformação real — não teatral.
  • Medo: Prometer magia e entregar fumaça (por isso, dashboards com dados reais, não storytelling vazio).
  • Estratégia: Mostrar resultado, não tecnologia. O dev vê o board atualizado e sente que é mágica. Sob o capô, é engenharia.

Secundário: O Criador (The Creator)

Valoriza a excelência artística. No Lin-Dev, isso se manifesta no design system obsessivamente construído, nas animações da gema, na tipografia Geist, na paleta deep purple. A ferramenta não é apenas funcional — é bonita. Porque desenvolvedores merecem ferramentas com a mesma qualidade do código que escrevem.

Buyer Personas

R
O Dev Solo Produtivo
"Rafael" — Full-stack dev, 26 anos, 3-5 anos de experiência
  • Cenário: Trabalha sozinho ou com 1-2 freelancers. Tem 3+ projetos paralelos.
  • Dor principal: Perde tempo mantendo Trello/Notion atualizados. Não tem visão clara do que está atrasado.
  • Motivação: Quer abrir o board e ver o status real, sem ter atualizado nada.
  • Gatilho de compra: Ver a dashboard mostrando onde cada projeto está. "Não preciso atualizar nada?"
  • Objeção: "Já tenho meu sistema no Notion." → "Seu Notion exige que você atualize ele manualmente. Todo dia."
  • Plano ideal: Free → Pro
C
O Tech Lead Pragmático
"Camila" — Tech Lead, 32 anos, squad de 4-8 devs
  • Cenário: Lidera um squad em startup. Gasta 1h+ por dia verificando status e montando reports.
  • Dor principal: Sprint meetings longas porque ninguém atualiza o board. Reports manuais toda sexta.
  • Motivação: Uma dashboard onde ela abre e vê: quem fez o quê, o que está atrasado, velocidade do time.
  • Gatilho de compra: ROI de 13× + a dashboard de sprint com dados reais. "Isso substitui minha planilha."
  • Objeção: "E se não tiver feature X do Jira?" → "Você usa 100% das features do Jira? Ou gasta tempo configurando features que ninguém usa?"
  • Plano ideal: Pro → Max
M
O Founder Técnico
"Marcos" — CTO/Co-founder, 35 anos, startup early-stage
  • Cenário: Squad pequeno (2-5 devs), queimando caixa. Faz code review, PM e product ao mesmo tempo.
  • Dor principal: Precisa de visibilidade total sem microgerenciar. Mais output com menos overhead.
  • Motivação: Dashboard de CTO: velocidade, prazos, carga, bloqueios. Tudo em uma tela. Dev Virtual como multiplicador.
  • Gatilho de compra: Ver o Dev Virtual abrindo 3 PRs enquanto o time faz code review.
  • Objeção: "Será que o código tem qualidade?" → "Testes passam. Code review é seu."
  • Plano ideal: Max (direto)

Manifesto

Nós acreditamos que a melhor gestão é a que você não percebe.

Que o board ideal não é o mais bonito — é o que se atualiza sozinho.

Que a melhor dashboard é a que responde antes de você perguntar.

Que o melhor processo é o que desaparece no fluxo.

Construímos o Lin-Dev porque cansamos de ferramentas que existem pra ser gerenciadas.

Ferramentas que pedem mais tempo do que devolvem.

Ferramentas que vivem numa aba que você nunca quer abrir.

Aqui, o board sabe o que está acontecendo.

As dashboards mostram prazos, bloqueios e velocidade — sem você montar nada.

O sprint fecha e o resumo já está lá.

E se você usa IA pra codar, ela lê tudo direto no terminal.

Sem copiar. Sem colar. Sem trocar de aba.

Mas o ponto não é a IA.

O ponto é o tempo que volta pro seu squad.

Isso não é o futuro.

É o presente que construímos juntos.

Aplicações por Canal

Website (Lin-Dev)

  • Background escuro. Nunca fundo branco.
  • Hero focado na proposta de valor da ferramenta (tempo devolvido, board automático).
  • Seção de dashboards com screenshots reais mostrando dados de sprint.
  • MCP e Dev Virtual aparecem como features, não como o pitch central.
  • CTAs sempre com Gem CTA.
  • Internacionalização: PT, EN, ES.

Redes sociais

  • Twitter/X: Conteúdo sobre dores reais (board desatualizado, sprint meeting de 1h). Screenshots de dashboards.
  • GitHub: README impecável. GIFs das dashboards e do MCP funcionando.
  • Discord: Comunidade, feedback loop, early adopters.
  • LinkedIn: Apenas ROI e case studies com dados. Nunca "post motivacional".

Produto (UI/UX)

  • Design System v1.0 é a referência absoluta.
  • Dashboards são first-class citizens: proeminentes, dados sempre frescos, leitura em <3 segundos.
  • Indicadores semáforos em cards: verde (no prazo), amarelo (risco), vermelho (atrasado).
  • Loading states com GemLoader (gema animada).
  • Micro-interações com spring easing.

Bloco K · Diretrizes de mídia

Fotografia & ilustração

O Lin-Dev tem uma posição clara: zero fotografia stock. A comunicação visual é construída com UI, código e a própria gema. Essa escolha mantém a identidade técnica e evita a estética genérica de SaaS.

K1

O que pode e o que não pode

Regra central: só usamos imagens que um desenvolvedor reconhece como "do seu mundo". Se precisaria explicar por que aquela foto existe, ela não entra.

Aceito
Mídia técnica
  • Screenshots reais do produto (dark-mode, sem chrome do browser)
  • Snippets de código em Geist Mono com syntax highlight da paleta
  • Gravações de terminal com o TerminalMock
  • Ícones em pixel-art respeitando o grid --px
  • Diagramas de arquitetura em linhas 1px tracejadas
Proibido
Evitar sempre
  • Fotografia stock de "pessoas colaborando em frente ao laptop"
  • Emojis como ilustração (exceto em quote/review de usuário real)
  • 3D renders genéricos, blobs abstratos, shapes fluidas alheias
  • Gradientes fora da paleta (qualquer coisa que não seja purple ou brand)
  • Ícones de outras bibliotecas (Heroicons, Font Awesome, Feather)
Zero fotografia stock UI + código + terminal + pixel-art Dúvida → pergunta: "um dev reconhece isso como familiar?"
